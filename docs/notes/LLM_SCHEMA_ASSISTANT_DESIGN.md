# LLM Schema Assistant Design

> Copyright (C) 2023-2026 Civic OS, L3C. Licensed under AGPL-3.0-or-later.

## Overview & Motivation

Civic OS's metadata-driven architecture creates a unique opportunity for LLM-assisted
schema design. Because the framework automatically generates CRUD views from database
metadata, an LLM only needs to produce SQL and metadata INSERTs — not application code.
This dramatically narrows the generation surface and makes output verifiable through
deterministic checks.

The LLM Schema Assistant system comprises three interconnected systems:

1. **Schema Assistant** — Generates SQL from natural language requests using assembled
   context (Integrator Guide, current schema state, few-shot examples)
2. **Safe Change Pipeline** — Database-backed review, approval, and apply workflow for
   any schema change (LLM-generated or manual)
3. **Model Evaluation Harness** — Benchmarks LLM providers against a standardized task
   suite with deterministic scoring

The Safe Change Pipeline is the shared foundation: the CLI produces changes that flow
through it, the eval harness uses it to apply and verify generated SQL, and the Angular
UI surfaces it for human review.

## Architecture Overview

Four phases of work, with shared infrastructure between them:

- **Phase 1 (CLI)** and **Phase 2 (Eval)** share context assembly, provider abstractions,
  SQL extraction, and safety validation
- **Phase 3 (Pipeline)** provides the database-backed approval workflow used by all other
  phases
- **Phase 4 (Angular + Go)** brings it into the Civic OS UI as an admin feature

```
Phase 1 (CLI)          Phase 2 (Eval)         Phase 3 (Pipeline)      Phase 4 (UI)

User Request --> CLI --> LLM --> SQL           +----------------+       Angular Page
                  |                            | pending_changes |       +-- Request Form
                  +-- Safety Check --> Output  | table           |<------+-- SQL Viewer
                                               |                 |       +-- Safety Display
Eval Tasks --> Runner --> LLM --> SQL --> PG   | approve/reject  |       +-- Approve/Reject
                 |                    |         | apply           |-------+-- History
                 +-- Score --> Report |         +----------------+
                                      |              ^
                                      +--------------+
                                      (eval uses same pipeline
                                       for apply + verify)
```

### Context Assembly Strategy

The context window is carefully budgeted to stay efficient across all providers:

| Layer | Size | Content |
|-------|------|---------|
| System prompt | ~3K tokens | Condensed rules in checklist form — required columns (`id BIGINT GENERATED BY DEFAULT AS IDENTITY`, `created_at`, `updated_at`, `display_name`), FK index requirement, RLS pattern, grant pattern, metadata INSERT patterns, NOTIFY requirement |
| Few-shot examples | ~2K tokens | 2-3 complete condensed examples showing all required pieces in execution order |
| Current schema state | ~1-5K tokens | Compact serialization of `schema_entities`, `schema_properties`, `schema_entity_dependencies`, `schema_rls_policies`, `schema_decisions` |
| Feature sections | ~500-2K tokens (on-demand) | Append relevant Integrator Guide sections based on keywords in the user request |
| User request | ~100-500 tokens | The natural language description of what to create or modify |
| **Total** | **~7-12K tokens input** | |

The system prompt is a condensed, LLM-optimized version of the Integrator Guide — not the
full document. It uses checklist formatting for maximum information density.

Feature sections are appended on-demand: if the user request mentions "status" or "workflow",
the Status Type System section is included. If it mentions "calendar" or "recurring", the
Calendar Integration and Recurring Time Slots sections are appended.

### LLM Output Format

The LLM is instructed to produce labeled SQL blocks with category markers. This enables
deterministic extraction and ordered execution:

```sql
-- [DDL] Table creation
CREATE TABLE public.work_orders ( ... );

-- [INDEXES] Index creation
CREATE INDEX idx_work_orders_status ON public.work_orders(status);

-- [STATUS] Status type configuration
INSERT INTO metadata.statuses (entity_type, name, ...) VALUES ...;

-- [CATEGORY] Category configuration
INSERT INTO metadata.categories (entity_type, name, ...) VALUES ...;

-- [METADATA] Entity and property metadata
INSERT INTO metadata.entities (...) VALUES ...;

-- [VALIDATIONS] Validation rules
INSERT INTO metadata.validations (...) VALUES ...;

-- [GRANTS] Permission grants
GRANT SELECT ON public.work_orders TO web_anon;

-- [RLS] Row level security policies
ALTER TABLE public.work_orders ENABLE ROW LEVEL SECURITY;
CREATE POLICY ...;

-- [PERMISSIONS] RBAC permission rows
INSERT INTO metadata.permissions (...) VALUES ...;

-- [FUNCTIONS] Computed fields and RPCs
CREATE OR REPLACE FUNCTION public.work_order_summary(...) ...;

-- [TRIGGERS] Timestamp and other triggers
CREATE TRIGGER ...;

-- [NOTIFY] Schema cache reload
NOTIFY pgrst, 'reload schema';

-- [ADR] Schema decision
SELECT create_schema_decision(...);
```

### Safety Validator

The safety validator is **hard-coded, not LLM-evaluated**. It classifies every SQL
statement into one of three risk tiers based on static analysis.

**Whitelist (safe operations):**

- `CREATE TABLE` / `ALTER TABLE ADD COLUMN` in `public`
- `CREATE INDEX`, `CREATE UNIQUE INDEX`
- `ALTER TABLE ADD CONSTRAINT`, `ALTER TABLE ENABLE ROW LEVEL SECURITY`
- `CREATE POLICY` on `public` tables
- `INSERT INTO metadata.*` (all metadata tables)
- `GRANT` on `public.*` tables/sequences/functions
- `CREATE OR REPLACE FUNCTION public.*`
- `CREATE TRIGGER` on `public` tables
- `NOTIFY pgrst, 'reload schema'`
- `SELECT create_schema_decision(...)`

**Blacklist (always rejected):**

- `DROP TABLE`, `DROP SCHEMA`, `TRUNCATE`
- Standalone DML on `public.*` data tables without accompanying DDL
- Raw `EXECUTE` / `DO $$` blocks that cannot be statically analyzed

**Review tier (allowed but requires human approval):**

- `ALTER TABLE DROP COLUMN` (when part of a data migration with revert script)
- DML on `public.*` tables when accompanied by DDL on the same table (data migration pattern)
- Dynamic SQL inside `CREATE FUNCTION` bodies

**Risk tiers:**

| Tier | Behavior |
|------|----------|
| `safe` | Auto-approvable, no human review needed |
| `review` | Allowed but requires explicit human approval |
| `dangerous` | Blocked entirely, cannot be approved |

### Key Interfaces

```typescript
interface SchemaAssistantProvider {
  name: string;
  supportsPromptCaching: boolean;
  generateSchema(context: AssembledContext, request: string): Promise<LLMResponse>;
  estimateCost(tokensUsed: TokenUsage): CostEstimate;
}

interface LLMResponse {
  rawResponse: string;
  sqlBlocks: CategorizedSQL[];
  reasoning: string;
  usage: TokenUsage;
  latencyMs: number;
  model: string;
  cost: CostEstimate;
}

interface TokenUsage {
  inputTokens: number;
  outputTokens: number;
  cacheReadTokens?: number;
  cacheWriteTokens?: number;
}

interface CostEstimate {
  inputCost: number;
  outputCost: number;
  cacheSavings?: number;
  totalCost: number;
}

interface CategorizedSQL {
  category: 'ddl' | 'indexes' | 'status' | 'category' | 'metadata' | 'validations' |
            'grants' | 'rls' | 'permissions' | 'functions' | 'triggers' | 'notify' | 'adr';
  sql: string;
  description: string;
  order: number;
}

interface SafetyResult {
  safe: boolean;
  warnings: { statement: string; reason: string }[];
  errors: { statement: string; reason: string }[];
  statements: { sql: string; risk: 'safe' | 'review' | 'dangerous'; reason: string }[];
}
```

### CLI Commands

```bash
# Generate schema SQL from a natural language request
civicos-schema generate \
  --postgrest-url http://localhost:3000 \
  --provider anthropic \
  --model claude-sonnet-4-20250514 \

# Validate an existing SQL file against safety rules

  --file generated-schema.sql \
  --postgrest-url http://localhost:3000

civicos-schema dry-run \
  --file generated-schema.sql \
  --db-url postgres://...
```

### Task Definition Format

Tasks are defined in YAML with explicit expected outputs and verification queries:

```yaml
id: create-work-order-entity
level: 2
title: "Create a Work Order entity with status and category"
starting_state: baseline
request: |
  Create a WorkOrder entity for tracking maintenance work orders.
  It needs a status workflow (Open -> In Progress -> Complete -> Cancelled),
  an assigned_to user field, a priority category (High/Medium/Low),
  a due_date, and a description text field.

expected_outputs:
  tables:
    - name: work_orders
      required_columns: [id, display_name, description, due_date, status, assigned_to, priority]
  status_types:
    - entity_type: work_order
      min_count: 4
  category_types:
    - entity_type: work_order_priority
      min_count: 3
  metadata_entities:
    - table_name: work_orders
  permissions:
    - { table: work_orders, permission: read, role: anonymous }
    - { table: work_orders, permission: create, role: authenticated }
  rls_policies:
    min_count: 4
  indexes:
    fk_columns_indexed: true
  notify_pgrst: true

verification_queries:
  - sql: "SELECT count(*) FROM information_schema.tables WHERE table_name = 'work_orders'"
    expected: 1
  - sql: "SELECT count(*) FROM metadata.statuses WHERE entity_type = 'work_order'"
    expected_min: 4
```

### Scoring Dimensions

All base scoring is **fully deterministic** — no LLM-as-judge for the core metrics.

| Dimension | Weight | Method |
|-----------|--------|--------|
| Syntax correctness | 15% | Parse with `@pgsql/parser`. Score = (parseable / total) x 100 |
| Schema correctness | 20% | Run verification queries against live DB. Score = (passed / total) x 100 |
| Metadata completeness | 20% | Count `metadata.*` rows vs expected. Score = (found / expected) x 100 |
| Permission correctness | 15% | Check grants + RLS + `permission_roles`. Score = (found / expected) x 100 |
| Convention adherence | 10% | Regex checks: snake_case? `ON CONFLICT`? `NOTIFY`? FK indexes? Score = average of 0/1 checks |
| Idempotency | 10% | Re-apply SQL to same DB. Score = 100 if clean, 0 if errors |
| Completeness | 10% | Requirements checklist from task definition. Score = (present / required) x 100 |

### Optional LLM-Judge Pass (Qualitative Layer)

On top of deterministic scoring, an optional frontier-model judge provides qualitative
analysis for aspects that resist rule-based evaluation:

- **Naming quality** — semantic equivalence (`assigned_user` is as valid as `assigned_to`)
- **Design quality** — coherence of the overall schema, appropriate feature usage
- **ADR quality** — depth of reasoning in the `schema_decision` call
- **Partial credit** — functionally correct but structurally different from expected output

The judge is **additive** — it explains and contextualizes deterministic results but never
overrides deterministic failures. A syntax error is a syntax error regardless of how
creative the naming is.

**Claude Code as Judge (Subscription-Based):** Instead of paying per-call API costs for
judging, use a `/eval-judge` Claude Code skill. The eval runner writes artifacts to
`tools/schema-eval/results/<run-id>/`, then invoking `/eval-judge results/<run-id>` reads
and analyzes them using subscription tokens. This provides zero marginal cost with full
codebase context available to the judge.

### Execution Flow

Each task uses a **one-shot prompt** — a single LLM call per task for reproducibility:

1. Start Docker Postgres with Civic OS baseline schema (tmpfs for speed)
2. Optionally apply `starting_state` if the task requires existing entities
3. Single LLM call with assembled context (system prompt + schema state + request)
4. Extract SQL blocks from response
5. Safety validate all statements
6. Apply to database
7. Run scoring dimensions
8. Generate report

### Task Suite

12 tasks across 4 difficulty levels:

| Level | Count | Examples |
|-------|-------|----------|
| Level 1 (Simple) | 4 | Add column, add validation, change status flow, add category |
| Level 2 (Medium) | 3 | Create entity with status, add M:M relationship, create entity with search |
| Level 3 (Complex) | 3 | Design complete system, add notification workflow, data migration (text to Category) |
| Level 4 (Expert) | 2 | Virtual entity with INSTEAD OF triggers, payment integration |

**Data Migration Tasks:** For tasks involving data migration (e.g., converting a text
column to a Category type), the eval runner seeds test data before the LLM call. The
context provided to the LLM includes `DISTINCT` values and counts — never raw row data.
The safety validator allows DML at review tier when accompanied by DDL on the same table.

### Cost Tracking & Reporting

Every LLM call records token usage and cost. The final report shows the Pareto-optimal
frontier of cost vs quality, helping identify the best model for production use:

```
| Model              | Composite | Cost/Task | Total |
|--------------------|-----------|-----------|-------|
| Claude Sonnet 4    |      92   |    $0.03  | $0.33 |
| GPT-4o             |      85   |    $0.04  | $0.44 |
| DeepSeek V3 (OR)   |      78   |    $0.005 | $0.06 |
```

## Phase 3: Safe Change Pipeline

The pipeline provides a database-backed workflow for reviewing, approving, and applying
schema changes. It is the shared foundation used by the CLI, eval harness, and future UI.

### Migration: v0-42-0-pending-changes

```sql
CREATE TABLE metadata.pending_changes (
    id SERIAL PRIMARY KEY,
    title VARCHAR(200) NOT NULL,
    description TEXT,
    source VARCHAR(50) NOT NULL DEFAULT 'manual',  -- 'assistant', 'manual', 'gui'

    -- SQL content
    statements JSONB NOT NULL,
    revert_statements JSONB,
    safety_analysis JSONB,
    syntax_valid BOOLEAN,
    dry_run_result JSONB,

    -- Workflow state
    status VARCHAR(30) NOT NULL DEFAULT 'pending',
    -- pending -> validated -> approved -> applying -> applied
    -- pending -> validated -> rejected
    -- pending -> validated -> approved -> applying -> failed

    -- People
    proposed_by UUID REFERENCES metadata.civic_os_users(id),
    proposed_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    validated_at TIMESTAMPTZ,
    reviewed_by UUID REFERENCES metadata.civic_os_users(id),
    reviewed_at TIMESTAMPTZ,
    review_note TEXT,
    applied_at TIMESTAMPTZ,
    apply_result TEXT,
    pre_apply_snapshot JSONB,

    -- Linked schema decision
    schema_decision_id INT REFERENCES metadata.schema_decisions(id),

    -- LLM provenance (NULL for manual changes)
    llm_provider VARCHAR(50),
    llm_model VARCHAR(100),
    llm_request TEXT,
    llm_tokens_used JSONB,
    llm_cost_usd NUMERIC(10,6),

    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT valid_status CHECK (status IN (
        'pending', 'validated', 'approved', 'rejected', 'applying', 'applied', 'failed'
    ))
);
```

### Supporting RPCs

| RPC | Purpose |
|-----|---------|
| `validate_pending_change(id)` | Parses SQL, runs safety analysis, sets `syntax_valid` and `safety_analysis`, transitions to `validated` |
| `approve_pending_change(id, note)` | Records reviewer, transitions to `approved` |
| `reject_pending_change(id, note)` | Records reviewer and note, transitions to `rejected` |
| `apply_pending_change(id)` | Captures pre-apply snapshot, executes statements in order, transitions to `applied` or `failed` |
| `revert_applied_change(id)` | Executes `revert_statements` if available, restores from snapshot |

### Permission Model

| Permission | Roles |
|------------|-------|
| `pending_changes:read` | editor, manager, admin |
| `pending_changes:create` | editor, manager, admin |
| `pending_changes:approve` | admin only |
| `pending_changes:apply` | admin only (or worker service account) |

The `apply_pending_change` RPC runs as `SECURITY DEFINER` because DDL statements
(CREATE TABLE, GRANT, etc.) require superuser or table-owner privileges that
authenticated users do not have.

### Backup on Apply

Before executing any statements, `apply_pending_change` captures:

1. **Metadata JSONB snapshot** — Current state of all `metadata.*` tables referenced by
   the change
2. **`pg_dump --schema-only`** — Full schema DDL for the `public` schema

This enables rollback even if the auto-generated revert statements are incomplete.

## Design Decisions

### Naming Convention: snake_case

All LLM-generated SQL uses `snake_case` for identifiers. Any PascalCase examples in
existing documentation are legacy artifacts. The system prompt explicitly requires
snake_case and the convention adherence scoring dimension checks for it.

### Evaluation: One-Shot Prompts Only

The eval harness uses a single LLM call per task — no multi-turn conversation, no
retries, no chain-of-thought follow-ups. This ensures:

- **Reproducibility** — same input produces comparable results across runs
- **Fair comparison** — all models get the same context and single chance
- **No conversation state** — eliminates a variable that differs across providers

The CLI supports multi-turn interaction for real usage. The eval measures atomic
capability. Prompt caching is valuable for CLI multi-turn sessions but irrelevant
for eval (each task is independent).

### Revert Strategy: Auto-Generated + LLM-Supplemented

Auto-generate reverts for operations with obvious inverses:

| Operation | Auto-Revert |
|-----------|-------------|
| `CREATE TABLE x` | `DROP TABLE IF EXISTS x` |
| `GRANT perm ON x TO role` | `REVOKE perm ON x FROM role` |
| `INSERT INTO metadata.entities` | `DELETE FROM metadata.entities WHERE ...` |
| `CREATE POLICY p ON x` | `DROP POLICY IF EXISTS p ON x` |
| `CREATE INDEX idx ON x` | `DROP INDEX IF EXISTS idx` |

For complex operations (functions with dependencies, multi-table migrations), the LLM
generates revert statements as part of its output. Missing reverts are flagged during
safety validation so reviewers know the change cannot be automatically rolled back.

### Backup Before Apply

Every `apply_pending_change` call captures a metadata JSONB snapshot and
`pg_dump --schema-only` before execution. This is defense-in-depth: even if revert
statements are wrong or missing, the pre-apply state can be restored manually.

### Data Safety

- **Schema metadata** is sent to LLMs (entity names, column types, constraints) — this
  is structural information, not user data
- **Row data is never sent** to LLMs
- For data migration tasks, provide `DISTINCT` values and counts (e.g., "status column
  has 4 distinct values: Open (45), Closed (30), Pending (12), Cancelled (3)") — this
  gives the LLM enough context to generate correct migration SQL without exposing records
- DML on `public` tables is only allowed at review tier and must be accompanied by DDL
  on the same table (data migration pattern)

## Maintenance

The Schema Assistant's effectiveness depends on its prompts staying in sync with the actual Civic OS patterns. When making changes to the core system, update the corresponding tool artifacts:

| Change Type | Files to Update |
|------------|----------------|
| New property type / custom domain | `tools/schema-assistant/prompts/system.md` (Custom Domains section), few-shot examples if relevant |
| New metadata table or column | `prompts/system.md` (relevant pattern section), `src/safety/validator.ts` (add to `METADATA_TABLES` whitelist) |
| New Integrator Guide section (e.g., new feature) | `src/context/guide-sections.ts` (add feature keyword mapping in `SECTION_MARKERS`) |
| Changed RLS/grant patterns | `prompts/system.md` (RLS and Grant Pattern sections), few-shot examples |
| New default role | `prompts/system.md` (Default Roles section), permission examples |
| New metadata RPC helper function | `prompts/system.md` (add to relevant pattern section) |
| Eval task format changes | `docs/notes/LLM_SCHEMA_ASSISTANT_DESIGN.md` (this file, Phase 2 section) |

The system prompt (`tools/schema-assistant/prompts/system.md`) is a **condensed** version of the Integrator Guide (`docs/INTEGRATOR_GUIDE.md`). It should stay concise (~3K tokens) — add only patterns the LLM needs to reproduce correctly, not exhaustive documentation.

## Future Considerations (Out of Scope)

These items are acknowledged but explicitly out of scope for the four phases above:

- **Sandbox / DB-fork for testing** — Spin up a throwaway database clone with
  representative (anonymized) data to test generated SQL against realistic conditions
  before applying to production
- **GUI Schema Builder** — Extend the existing Schema Editor ERD with drag-and-drop
  entity creation that generates SQL through the assistant pipeline
- **Continuous eval on new model releases** — Automated re-runs of the eval suite when
  new model versions are released, with regression alerts if scores drop
