# Eval Run: 2026-03-28 — Invoice Entity

## Request

Add an 'invoices' entity for tracking client invoices. It needs a client FK, amount (money), invoice_date (date), a status workflow (Draft, Sent, Paid, Overdue, Cancelled), and a notes text field.

## Target Instance

- **Existing schema**: clients, projects, time_entries, contact_log, design_docs, scheduled_contacts
- **Context size**: ~64K input tokens
- **Complexity level**: 2 (medium — new entity with status workflow and FK to existing table)

## Rounds

Three rounds were run. Between rounds, the system prompt was improved based on observed failures.

### Round 1 — Initial prompt (12 models)

Tested all available DigitalOcean models to establish baseline quality.

**Eliminated:**
- `kimi-k2.5` — Produced 0 SQL blocks despite 8,192 output tokens. Did not follow labeled block format.
- `llama3.3-70b-instruct` — Missing `display_name` column (blocker), `SERIAL` instead of `BIGINT GENERATED BY DEFAULT AS IDENTITY`, Overdue marked terminal, hit 4K output limit.
- `alibaba-qwen3-32b` — API error (failed to generate).

### Round 2 — Expanded model set (7 models)

Added Anthropic frontier models and new open-weight models via DigitalOcean.

### Round 3 — Improved prompt (7 models)

System prompt updated with: `created_by` required column, sensitive entity access rules, `status_key` values, `status_entity_type` configuration, `manager` role in examples, money CHECK pattern, no-stray-markdown rule.

## Key Findings

1. **Prompt engineering lifted all models 5-11 points** between rounds 1/2 and round 3. The biggest gains came from explicit rules for sensitive data access, required columns, and `status_key` values.

2. **No model is deploy-ready without review.** Even the top scorer (GPT 5.4, 98) has minor issues. The stray markdown artifact in Anthropic models would cause parse errors. All outputs should go through the safe change pipeline.

3. **Cost is flat (~$0.35-0.43)** because input tokens (~64K) dominate. The price difference between models is negligible for this task size. Speed and quality are the differentiators.

4. **GPT 5.4 is the best cost/quality/speed combination** — highest score, cheapest, second-fastest. GLM-5 is the best open-weight model at 97.

5. **The CC Opus baseline (95) was surpassed by 5 API models after prompt improvements.** This validates the approach: invest in prompt quality rather than relying on the most expensive model with full codebase context.

6. **Sensitive data handling was the hardest pattern to teach.** Round 1/2: only 2/12 models restricted anonymous access. Round 3: 7/7 did after adding an explicit rule. Domain judgment can be encoded as rules.
